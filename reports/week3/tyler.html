Tyler's Week 3 Individual Report

What were your concrete goals for the week?

My concrete goals for the week were to
  1. Get boost::asio included in our CMake
  2. Learn boost::asio
  3. Start brainstorming / implementing structs to represent our packet data
  4. Set up a basic TCP echo client/server using boost::asio
  5. Get packets sent across the network on the lab computers

What goals were you able to accomplish?

Working with Edward, we were able to accomplish all of the above goals except number 5. For number 5, we were only able to get it working across localhost within one lab computer. However, on Sunday we realized that we were not using a port that was allowed through the firewall, and at the time of writing we haven’t been able to test it in the lab again, so we think it should work because we were able to successfully send packets across other networks.

In addition, I was also able to implement the following features:

  1. Helper functions to create/serialize packets using boost::serialize. I ended up using boost::serialize because we already were getting it for free since we included boost for asio, and it simplified a lot of the logic of packet structure. Also, we likely were going to want it anyways to be able to serialize the GameState class and send it across the network.
  2. A Lobby Discovery/Broadcast feature for clients/servers so that we don’t have to manually figure out the IPs of the machines we are working on. In theory this works, but we have to verify it works on the lab network, and we probably will have this disabled until we have a UI and can implement user interactivity through it.
  3. A more advanced Client/Server class structure that allows for “Sessions” to be stored that represent one Client <-> Server connection. This session is used on both the client and server side, and has helper functions to simplify the process of sending and receiving our packets across the network. I worked more on the server side of things, and Edward worked more on the Client side of things.
  4. A Config file that we can use to configure network settings and other options for the program.

Unrelated to networking, I also helped with setting up the github repository and CI for our website, since as a group we decided we wanted to move away from wordpress and make our own custom website.

What did you learn this week, if anything (and did you expect to learn it?)

The main thing I learned was how to use boost::asio, and I also reinforced a bunch of C++ knowledge that I vaguely knew, but was shaky on. This came from a couple rough debugging sessions with asio where I wasn’t correctly managing the lifetimes of the data my buffers were referencing and they were getting deallocated before actually being sent across the network—which made me appreciate Rust more. I also got more comfortable with asynchronous programming because asio lets you make extensive use of asynchronous methods where you pass in callback functions.

I expected to learn about the intricacies of boost::asio, but one thing I didn’t expect to learn were a bunch of other boost related libraries and utilities. I didn’t end up using all of them, but I didn’t realize how much functionality boost provides. Boost::serialization was especially helpful and easy to use. I think it definitely isn’t as efficient as doing it manually and sending the raw bytes of our packet structs, but it lets us not worry about network byte ordering and struct packing and other lower level concerns. If it ends up being too much overhead then we might have to switch away from it, but we would still probably use it for the packet that sends the entire GameState.
