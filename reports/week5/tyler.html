    If the week went differently than you had planned, what were the reasons?

    • At a high level things went as I expected, but there were a lot of smaller implementation level details that I hadn’t planned needing to solve.
    • There were an obnoxious amount of bugs dealing with rendering for Sprites that took the entire week on/off to solve. David and I kept alternating taking cracks at it, each finding new bugs and problems but not the root cause. It was very demoralizing because you would have an empty screen being rendered, then you would find a definite bug and fix it, only to find that you were still rendering an empty screen.
    • I thought I had most of the high level design done in the previous week for the GUI library, but I realized quickly that I hated everything and it was all bad, so throughout the entire week I slowly tweaked things as I had to actually start using my interfaces and things annoyed me. I still kind of hate it, but I know it would be a herculean effort to redo everything and make it completely generalized for anything you might want to do, and what I have right now should be able to do ~90% of anything we would want to do, and if it can’t then we will just have to think of a creative way to work around it.

    What are your specific goals for the next week?

    • There are a couple more UI things that I need to touch up on, namely finalizing the code for positioning Sprites so that we can arbitrarily render them to coordinates on the screen, instead of them just being in the center all overlapping.
    • In addition, I want to potentially start looking into optimizing the networking code slightly so that the Server only sends gamestate updates for objects that have changed. Therefore it won’t have to send down the information for each and every “Wall” object on every frame, which should save on a lot of bandwidth.
    • The motivation for this came around after we ran into our first networking related bug where we had set aside a buffer of 10,000 bytes for incoming packets, which we finally exceeded once our mazes got larger than ~60 walls. This optimization won’t solve this problem because the initial load will still need to handle all the objects, but it still will be nice to have.
    • However, the networking optimizations may not be the highest priority so after our Wednesday meeting at 7:30pm (which will be after this report is turned in) if we determine there is a more pressing task (like perhaps combat / traps / some other game system) that somebody needs to do, then I may put off the networking optimization.

    What did you learn this week, if anything (and did you expect to learn it?)

    • I learned a lot about the OpenGL rendering pipeline. Before I was completely 100% phoning it in by copy pasting code from online, but now after being forced to parse it and understand what is going on in order to debug rendering problems I am starting to understand how things work.
    • I also just got a lot of experience designing an API and then having to tweak it as I realized my initial assumptions about how I would use it turned out to be wrong.

    What is your individual morale (which might be different from the overall group morale)?
